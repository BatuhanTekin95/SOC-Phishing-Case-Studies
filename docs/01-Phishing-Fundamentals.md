# Phishing Fundamentals

## Purpose

This document is my reference for phishing triage and email analysis. It covers the parts of an email that matter during an investigation, the checks I perform, and the limits of common email security controls.

The main lesson is simple: one suspicious indicator is rarely enough. A reliable verdict comes from correlating sender identity, trusted headers, authentication results, URLs, attachments, endpoint activity, and user context.

## Email Address and Message Identity

An address such as `analyst@example.com` contains a local part (`analyst`) and a domain (`example.com`). During triage, I compare several identities rather than trusting the visible From field:

- **From:** Address displayed to the recipient.
- **Reply-To:** Address that receives a reply. A mismatch may be suspicious but can also be legitimate.
- **Return-Path:** Envelope sender used for delivery status messages and SPF evaluation.
- **Message-ID:** Message identifier normally generated by a mail system. An unusual value is a clue, not proof of forgery.

Display names can be changed easily, and a lookalike domain can pass email authentication for its own domain. I therefore inspect the complete address and domain spelling.

## How Email Is Delivered

- **SMTP** transfers a message from a mail client to a sending server and between mail servers.
- The sending mail server uses DNS MX records to identify the recipient domain's mail server.
- The recipient server accepts or rejects the SMTP transaction and, if accepted, places the message in the recipient's mail system.
- **IMAP** and **POP3** are client access protocols used later to read or retrieve stored messages. They are not part of the server-to-server delivery decision.

### Reading the Received Chain

Each mail relay normally prepends a `Received` field. This means the newest hop is at the top and the oldest visible hop is near the bottom.

I read the chain from the bottom upward, but I only trust entries added at or above the first mail system controlled by the receiving organization. Lower entries may have been supplied by the sender and can be forged. A source IP should therefore be supported by the trusted gateway's connection data where possible.

## Common Phishing Types

- **Phishing:** Broad deceptive messaging intended to steal information or deliver malware.
- **Spearphishing:** A targeted message tailored to a person or organization.
- **Whaling:** Spearphishing focused on executives or other high-value targets.
- **Malspam:** Bulk email used to distribute malicious links or files.
- **Smishing:** Phishing delivered through SMS.
- **Vishing:** Social engineering conducted through voice calls.

Common themes include invoices, account problems, shared documents, deliveries, password resets, and urgent requests from management. These themes create pressure, but urgency alone does not make a message malicious.

## Triage Workflow

### 1. Establish Scope

- Who received or reported the message?
- Was it delivered to other mailboxes?
- Did anyone click, reply, enter credentials, or open an attachment?
- What was the delivery time and time zone?
- Is the message still present in the environment?

### 2. Preserve Evidence

- Export the original message as `.eml` or the platform's native format.
- Record the message trace or gateway identifier.
- Calculate hashes for attachments before analysis.
- Keep the original file unchanged and work from a copy.

### 3. Review the Message

- Compare From, Reply-To, and Return-Path.
- Check the subject, tone, request, and business context.
- Inspect link text and actual destinations.
- Look for misleading extensions, archives, or password-protected files.
- Treat logos and signatures as presentation, not proof of identity.

### 4. Review Trusted Headers

- Identify the first trusted `Received` hop and connecting IP.
- Review trusted `Authentication-Results` added by the receiving system.
- Check SPF, DKIM, DMARC, and alignment.
- Review gateway verdicts, spam scores, and delivery action.
- Note forwarding or mailing-list behavior that may affect authentication.

### 5. Enrich Indicators

- Defang URLs before documenting or sharing them.
- Check domain age, passive DNS, reputation, and hosting context.
- Search file hashes in multiple intelligence sources.
- Distinguish a legitimate provider from an abusive tenant using that provider.
- Record the time of each reputation lookup because results can change.

### 6. Correlate Endpoint and Identity Activity

If interaction occurred, email analysis is only the starting point. I check:

- Browser and proxy activity
- Process creation and command lines
- File creation and execution
- Authentication events and MFA changes
- Mailbox rules and forwarding settings
- Network connections and DNS requests
- Related alerts on the same user or host

## Email Authentication

### SPF

SPF allows a domain to publish which systems are authorized to use its envelope sender identity. A result of `fail`, often produced by an `-all` qualifier, means the evaluated sending IP was not authorized by that SPF policy.

SPF does not verify the visible From address and does not force every receiver to take the same delivery action. Forwarding can also affect SPF results.

### DKIM

DKIM uses a cryptographic signature to validate selected headers and message content for the signing domain. A valid signature shows that the signed parts were not changed after signing and that the signer had access to the domain's private key.

DKIM does not prove that the message is safe. An attacker can send a DKIM-valid message from a domain they control or from a compromised legitimate account.

### DMARC

DMARC evaluates whether the visible From domain aligns with a passing SPF result or a passing DKIM signature. A message can pass DMARC when at least one aligned authentication method passes.

Published policies request handling for DMARC failures:

- `p=none`: collect reports and monitor; no requested enforcement action.
- `p=quarantine`: request suspicious treatment, such as spam placement.
- `p=reject`: request rejection of messages that fail DMARC.

The receiving organization still applies its local policy. `p=none` does not guarantee delivery, and `p=quarantine` does not prove where a specific message was placed.

### ARC and S/MIME

- **ARC** records authentication observations across intermediaries and can help a receiver evaluate forwarded mail. It does not make a message trustworthy by itself.
- **S/MIME** can provide message signing and encryption using user certificates. Its value depends on certificate validation, private-key protection, and organizational trust policy.

## Fields I Check

| Field | Investigation value | Important limitation |
| --- | --- | --- |
| Authentication-Results | SPF, DKIM, and DMARC results | Trust only when added by a known receiving system |
| ARC-Authentication-Results | Authentication observed before forwarding | ARC chain and sealer still require evaluation |
| Received-SPF | SPF result and evaluated identity | May reflect only one stage of delivery |
| DKIM-Signature | Signing domain and covered fields | A valid signature does not mean safe content |
| Return-Path | Envelope sender used by SPF | May legitimately differ from visible From |
| Reply-To | Destination for replies | A mismatch is contextual, not automatically malicious |
| Received | Relay path and connecting systems | Untrusted entries can be forged |

## URL and Attachment Safety

I do not open suspicious links or files on a production workstation. Analysis should take place in a controlled environment with appropriate network restrictions.

For URLs, I record the original value, create a defanged copy for reporting, inspect redirects, and check whether the destination requests credentials or downloads content.

For attachments, I compare:

- Displayed and actual filename extensions
- File signature or magic bytes
- MIME type and archive contents
- SHA-256 hash and reputation
- Static indicators such as macros, scripts, URLs, and embedded objects
- Sandbox behavior when detonation is authorized

Opening a PDF, Office document, or archive does not always execute code automatically. Execution depends on the file contents, software behavior, vulnerabilities, macro settings, and user actions.

IOCs should include context such as first seen, last seen, source, confidence, and expected expiration. A shared hosting IP or common service should not be blocked without checking possible business impact.

## MITRE ATT&CK Mapping

ATT&CK mapping should describe behavior supported by evidence, not every possible outcome of a phishing email.

| Observed behavior | Technique | ATT&CK ID |
| --- | --- | --- |
| Malicious attachment delivered by email | Spearphishing Attachment | T1566.001 |
| Malicious link delivered by email | Spearphishing Link | T1566.002 |
| User opened a malicious file | User Execution: Malicious File | T1204.002 |
| User opened a malicious link | User Execution: Malicious Link | T1204.001 |

## Selected SMTP Status Codes

| Code | General meaning | Investigation use |
| --- | --- | --- |
| 220 | Service ready | SMTP session can begin |
| 250 | Requested action completed | Server accepted the current SMTP command |
| 421 | Temporary service problem | Retry may occur later |
| 450 / 451 / 452 | Temporary failure | Message may remain queued |
| 550 / 553 / 554 | Permanent failure or policy rejection | Review the enhanced code and gateway logs |

The enhanced status code and mail gateway logs provide more context than the three-digit response alone.

## Analyst Checklist

- [ ] Preserve the original message and record its identifiers.
- [ ] Confirm recipient, delivery time, and user interaction.
- [ ] Compare From, Reply-To, and Return-Path.
- [ ] Identify the first trusted Received hop.
- [ ] Validate SPF, DKIM, DMARC, and alignment.
- [ ] Extract and defang links.
- [ ] Hash and identify attachments safely.
- [ ] Enrich IOCs with time and source recorded.
- [ ] Search for related messages, users, and hosts.
- [ ] Separate confirmed evidence from assumptions.
- [ ] Record containment actions and remaining gaps.

## Conclusion

A good phishing investigation is based on correlation. Authentication results, reputation scores, or a suspicious filename can support a verdict, but each has limitations. My goal is to document what the evidence proves, what it only suggests, and what additional data would be needed to close the remaining gaps.
